逃离鸭科夫让我第一次尝试了c#的mod开发，并学到了不少东西。在这里记录一些有趣的经历。

1.资料收集#

Unity#

Unity使用两种脚本后端，分别是mono（JIT）和IL2CPP（AOT）。mono是Unity的默认后端，也幸而是鸭科夫使用的后端，这让整个mod开发过程变得简单。

Mono#

Mono是.NET Framework的跨平台实现，定位类似openJDK；Unity使用Mono的一个分支作为脚本后端

.NET Framework#

.NET Framework是windows-only的软件框架（主要语言为C#），其定位类似oracle JDK，但已经被Microsoft的新跨平台开源实现.NET (Core)取代

.NET Standard#

.NET Standard是针对多个.NET实现推出的统一api规范；鸭科夫使用的是常见的.NET Standard 2.1

2.使用工具#

  • dotnet sdk（当然）
  • Harmony（hook框架）
  • dnSpy（更好的反编译）
  • Unity Explorer（可以视觉定位结构）

3.开发趣事#

具体开发过程没什么好讲的，记录一些有趣的部分：

感悟与知识#

鸭科夫的可执行文件完全没有任何混淆，dnspy能反编译出非常reasonable的代码（dotnet本身也有反编译功能，但质量不如dnspy）

.net的开发体验绝对是顶级的，对于我这样的小项目，msbuild用起来很无脑，nuget很好用，而且居然可以直接include游戏的dll…

C#用起来和c++基本一致，但少了很多痛点（得益于gc），实现一个功能基本不用考虑“茴的四种写法”（也或许是还没遇到其他写法？）

反射简直是史诗级加强！（libffi：在想我的事？）

如果一个方法有多个重载，就需要向HarmonyPatch传入一个常量Type数组以指定重载版本；坑的地方来了，MakeByRefType()不能在编译期决定，所以写：

[HarmonyPatch(typeof(ItemRepairView), "CalculateRepairPrice", 
    new Type[] { 
        typeof(Item), 
        typeof(float).MakeByRefType(), 
        typeof(float).MakeByRefType(), 
        typeof(float).MakeByRefType() 
    }
)]

会报错，只能在TargedMethod动态patch（用AccessTools）

patch lambda没那么直接，查看il可以发现lambda方法实际在一个名为<>c的内部类中（如果有捕获名字会更长）需要AccessTools.Inner先取得内部类

AccessTools是类型反射，而Traverse就是实例反射，最方便的是能够调用实例方法来实时影响实例变量（需要为目标示例创建traverse实例）

hook与逻辑部分是云淡风轻岁月静好，ui部分的实现则完全是地狱： (skill issue)
