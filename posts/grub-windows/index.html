使用windows安装grub与使用linux基本无异，这里记录一些实操时遇到的小坑

GNU grub安装套件#

gnu提供了windows构建版，使用方法与linux版区别不大；如果已经熟悉了grub的安装方法，可以跳到差异部分。

grub的构建与工作机理#

grub采用模块化设计，以便适应各种各样的启动需求。在安装套件中可以看到大量.mod文件，这些模块实现了不同硬件接口/分区/文件系统驱动，或提供显示图片/视频，命令行等功能，构建时可以按需选择打包进主程序。

使用grub-install安装时，grub-install首先调用grub-probe探测目标文件系统信息，接着使用grub-mkimage选择必要的模块打包。实际安装时，其余太大没打包进主程序的模块和资源资源文件将会存放在boot分区/boot/grub或用户指定的位置。

在传统bios系统中，grub的boot.img存放在主引导记录（MBR），它会做一些准备工作并跳转到grub的主程序core.img（通常存放在mbr的后续空扇区中，如果空间不够则需要置于独立的boot分区中）

在uefi系统中，grub作为efi程序安装在efi系统分区（esp）；uefi固件扫描esp，根据nvram中的启动项加载对应的efi可执行文件（如grubx64.efi）。

接下来，grub会根据构建grub时指定的路径加载配置文件（grub.cfg）和必要的模块，并按照配置与用户指令完成引导过程。

基本安装#

grub-install基本使用方法在这里；
当安装目标为efi时，--efi-directory指定esp分区的挂载点，grub-install会在这里的/efi/grub安装引导文件；
--boot-directory指定grub的配置文件和模块存放位置，会安装在指定位置的/grub目录中。

需要注意的是，如果不指定--boot-directory，grub-install会默认将配置文件和模块存放在工具所在分区根目录下的/grub中；这在windows下会成为问题。

window下的差异#

挂载esp分区#

在linux中，esp分区通常会挂载到/boot/efi下，但windows并不会自动挂载esp分区，因此在安装前我们得先手动挂载：

diskpart          // 运行diskpart工具
list disk         // 列出所有磁盘
select disk 0     // 选中目标磁盘
list part         // 列出所有分区
select part 1     // 选中esp分区
assign letter=M   // 为分区分配盘符

安装grub#

接下来我们就可以运行grub-install命令了：

grub-install --target=x86_64-efi --efi-directory=M: --boot-directory=M:/boot

安装完成后，我们需要编辑启动项：

bcdedit /enum firmware                                  // 列出所有启动项
bcdedit /set {bootmgr} path \EFI\grub\grubx64.efi       // 设置grub为默认启动项
// 注意{bootmgr}的partitions是否正确！

创建grub.cfg#

最后，我们需要创建grub.cfg放在/grub目录下

简单写一个windows的：

    set timeout=5
    set default=0

    menuentry 'Windows EFI Boot Manager' {
        set gfxpayload=2560x1600
        insmod chain                        //加载chainloader模块
        insmod fat
        set root=(hd0,gpt1)
        chainloader /efi/Microsoft/Boot/bootmgfw.efi
    }

grub，启动！#

剧情发展到这里，不出意外的话就要出意外了（

如果使用的是gnu官方的grub-2.12-for-windows，你将会启动到grub并看到：

symbol '_stack_chk_guard' not found

十分扎心的黑底白字

在这组信件存档可以看到，grub2win的作者也遇到了这个问题，几位大佬讨论一番，最终由Daniel的一句

I will take a look at this issue in the following days...

彻底告终。

经过我的测试，旧版本的x86与x64，2.12版的x86版均不会出现这个问题，在linux下编译的grub也不会出现这个问题

经过某种机缘巧合，我用linux编译的grub模块替换掉grub-2.12-for-windows的模块后，竟然就ok了……

所以最简单的方案：去debian下载grub2.12的deb包，用内部的x86_64-efi目录覆盖掉windows版有问题的模块，然后重新运行grub-install即可。

我也没仔细研究是什么导致了这个错误，或许是mingw的stack protector ABI问题…？

grub美化#

安装成功后奖励一下自己

Gorgeous-GRUB索引了许多有趣的grub主题，安装方式也很简单，将theme目录直接拷贝到/grub/themes下，然后在grub.cfg中添加：

set theme=/boot/grub/themes/dir-of-your-theme/theme.txt
