从ISO镜像文件启动linux发现的ntfs驱动问题

2025-09-212 min read (868 words)
目录
  • 1.前置知识
  • 2.发现问题
  • 3.解决方法

许多linux发行版都支持从iso镜像启动，本质是在initramfs阶段创建了一个loop块设备并从中启动内核，一次实践让我发现了ntfs3的一个小问题.

1.前置知识#

loop的作用是将文件映射为块设备，将块层I/O请求转换为文件操作。具体的映射在这个函数中实现：

loop.c#L415 

static int do_req_filebacked(struct loop_device *lo, struct request *rq)
{
	struct loop_cmd *cmd = blk_mq_rq_to_pdu(rq);
	loff_t pos = ((loff_t) blk_rq_pos(rq) << 9) + lo->lo_offset;

	switch (req_op(rq)) {
	case REQ_OP_FLUSH:
		return lo_req_flush(lo, rq);
	case REQ_OP_WRITE_ZEROES:
		/*
		 * If the caller doesn't want deallocation, call zeroout to
		 * write zeroes the range.  Otherwise, punch them out.
		 */
		return lo_fallocate(lo, rq, pos,
			(rq->cmd_flags & REQ_NOUNMAP) ?
				FALLOC_FL_ZERO_RANGE :
				FALLOC_FL_PUNCH_HOLE);
	case REQ_OP_DISCARD:
		return lo_fallocate(lo, rq, pos, FALLOC_FL_PUNCH_HOLE);
	case REQ_OP_WRITE:
		return lo_rw_aio(lo, cmd, pos, ITER_SOURCE);
	case REQ_OP_READ:
		return lo_rw_aio(lo, cmd, pos, ITER_DEST);
	default:
		WARN_ON_ONCE(1);
		return -EIO;
	}
}

2.发现问题#

测试使用mkfs.ext4 -F创建了5g的img，再用losetup -fP创建loop报错：

[xxx.xxxxxx] operation not supported error, dev loop0, sector xxxxxx op 0x9:  
(WRITE_ZEROES) flags 0x800 phys_seg 0 prio class 2

这对应的是块请求REQ_OP_WRITE_ZEROES，在loop中被转换为FALLOC_FL_ZERO_RANGE或FALLOC_FL_PUNCH_HOLE（是否要释放）

然后我们来到ntfs3的ntfs_fallocate，就会发现它只处理了punch hole，没有zero range…

但是事情没那么简单，如果继续往下读FALLOC_FL_ZERO_RANGE的处理逻辑，会发现它调用了一个叫ntfs_zero_range的函数，这看起来正是我们想要的zero_range实现，但其只作为punch hole的辅助函数，没有暴露出来，这会是为什么呢？？？

合理的解释是：zero range在ntfs上实现并不容易。
对于一些linux文件系统比如ext4，拥有unwritten extent机制：zero range时可以不真正写0，只修改extent metadata，读时直接返回0；
而ntfs没有对应的机制，想实现zero range只能写0，但这就失去zero range的意义了，不如直接write；如果硬要支持，只能承受低效or魔改的代价；
补充信息：patchwork

3.解决方法#

其实解决办法也很简单：换成不那么依赖zero range的fs就好了！
btrfs这样的CoW文件系统就不会有这样的问题；最后成功boot iso：

menuentry 'archiso test' {
    loopback loop (hd1,gpt3)/dl/mirror/archlinux-2025.02.01-x86_64.iso
    linux (loop)/arch/boot/x86_64/vmlinuz-linux img_dev=UUID=xxxxxxxxxxxx img_loop=dl/mirror/archlinux-2025.02.01-x86_64.iso earlymodules=loop
    initrd (loop)/arch/boot/x86_64/initramfs-linux.img
    boot
}
